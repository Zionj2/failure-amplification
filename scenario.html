Failure Amplification

A concrete cascade: small issues → compounding pressure → outage.

Concept Scenario Takeaways

Scenario

The “minor slowdown” that becomes a full outage

This is a deliberately realistic pattern: nothing “explodes” at first. The system degrades, people compensate, the compensations add load, and the loop accelerates.

Setting

A small web app has a database, an API, and a frontend. Traffic is steady. The team is small. Monitoring exists, but alerts are noisy and often ignored.

The cascade

  1. 1

    Small change increases latency

    A new feature adds a database query that’s slightly inefficient. Requests still succeed, but average response time rises. It’s noticeable, not alarming.

    Amplifier: delay (damage accumulates quietly).

  2. 2

    Time-outs start, but only occasionally

    A small fraction of requests now time out during busy moments. Users refresh. The refreshes add traffic. Most requests still succeed, so the problem “feels intermittent.”

    Amplifier: feedback loop (retries create load).

  3. 3

    Retries and queues hide the severity

    Client-side retry logic and server queues smooth over failures. Metrics show “success,” but the system is now spending more work to achieve the same results.

    Amplifier: masking (success rate hides cost).

  4. 4

    A workaround makes things worse

    Someone disables a slow safeguard (or increases a timeout) to make the app “feel faster.” The app looks better briefly, but requests now sit longer, holding resources, increasing contention.

    Amplifier: workarounds (risk moves and grows).

  5. 5

    Coupling spreads the pain

    As the database slows, API instances pile up waiting. CPU and memory rise. Autoscaling adds more instances, which add more connections to the already-struggling database.

    Amplifier: coupling (helpful scaling becomes load).

  6. 6

    Alert fatigue delays response

    Alerts have fired before for “minor” spikes, so the team hesitates. Meanwhile, the retry loop continues. The system is now operating near its limit with no slack.

    Amplifier: human delay (signal ignored until late).

  7. 7

    The loop snaps into runaway failure

    Latency increases → more retries → more load → more latency. Eventually, queues fill, timeouts surge, and the database starts rejecting connections. The app appears “down.”

    Amplifier: positive feedback (self-accelerating).

  8. 8

    Recovery is harder than prevention

    Even after fixing the original query, the system has to drain backlogs, clear caches, and unwind retries. The “small” failure required “big” recovery work.

    Amplifier: inertia (built-up load takes time to unwind).

Takeaways

Watch cost, not just success

A system can be “mostly successful” while becoming dramatically more expensive per request. Rising work-per-result is an early warning.

Retries are a hidden amplifier

Retries improve user experience until they become the main source of load. Make retries visible and bounded.

Slack prevents cascades

Systems fail when there’s no margin: time margin, capacity margin, attention margin. Slack is not waste; it’s stability.

That’s the whole point: the “cause” was small, but the system conditions made it amplify.
