Scenario

Coupling Cascade

One dependency hiccup becomes everybody’s problem because boundaries aren’t real boundaries.

Patterns involved

Hidden Coupling Queue Collapse No Slack

Reference: Hidden Coupling

The cascade

  1. 1

    A shared dependency slows

    A service (auth, DB, cache) slows slightly. Nothing fails outright yet.

    Amplifier seed: shared dependency.

  2. 2

    Multiple components pile onto the same choke point

    Several parts of the system all depend on that service. Requests start stacking up everywhere.

    Amplifier: coupling (impact spreads).

  3. 3

    Queues grow across boundaries

    Each component queues work while waiting. Queue depth increases, and timeouts start appearing.

    Amplifier: queue collapse.

  4. 4

    No slack means everything tips together

    With little margin, the system can’t isolate the blast radius. Several services degrade at once.

    Amplifier: no slack.

  5. 5

    Recovery becomes coordination-heavy

    Fixing the dependency is necessary but not sufficient. You must drain backlogs and stabilize multiple services.

    Key idea: coupling increases both failure spread and recovery complexity.

How to interrupt this cascade

Isolation & fallbacks

Make components fail independently. Build graceful degradation paths.

Backpressure

Bound queues and reject work early so waiting doesn’t metastasize across services.

Margin

Slack buys time and prevents “everything fails at once.”
